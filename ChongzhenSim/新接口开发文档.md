崇祯模拟器 - 人才与问政玩法AI友好型接口设计文档

1. 文档概述

1.1 文档目的

本文档针对崇祯模拟器的「人才玩法」和「问政玩法」，设计AI友好型接口，明确接口调用规范、参数定义、响应格式及AI交互逻辑，支撑游戏中AI驱动的人才动态演化、朝堂问政实时交互功能，确保接口适配大语言模型（LLM）的语义解析、逻辑推演能力，满足“文字即权柄”的核心玩法需求，同时保障接口易用性、可扩展性，适配游戏“千人千局”的动态模拟特性。

1.2 适用范围

本文档适用于崇祯模拟器开发团队（前端、后端、AI算法），作为人才玩法、问政玩法的接口开发、联调、测试依据，同时为AI模型接入提供清晰的交互规范，覆盖玩家与AI人才、AI大臣的全场景交互，包括人才招募、培养、问政议政、诏书下达与反馈等核心流程。

核心原则

- AI友好：接口参数设计贴合LLM语义解析习惯，支持自然语言输入（如玩家自定义诏书、问政话术），响应结果适配AI动态推演需求，保留足够的灵活度以支撑“非固定脚本”的交互逻辑。

- 拟真适配：贴合明末历史背景，接口参数、响应数据贴合朝堂机制、人才特质，还原孙承宗、徐光起、魏忠贤等历史人物的性格与立场，支撑AI对人物行为的精准模拟。

- 高效低延迟：针对AI实时交互场景（如问政对话、人才响应），优化接口性能，确保响应延迟≤500ms，适配游戏实时推演需求，避免影响玩家沉浸感。

- 可扩展性：接口设计预留扩展字段，支持后续人才品类、问政议题、AI交互模式的迭代，适配游戏国策规划、灾情、动乱等多维度场景的扩展需求。

- 数据一致性：确保接口调用后，人才状态、朝堂态势、国策执行结果等数据实时同步，支撑AI对全局态势的连贯记忆与逻辑推演，避免“记忆断层”。

2. 通用规范

2.1 接口基础信息

- 接口协议：HTTP/HTTPS

- 请求方式：POST（优先）、GET（适用于简单查询）

- 数据格式：JSON（编码UTF-8），支持自然语言文本字段的灵活输入

- 字符编码：UTF-8

- 基础路径：/api/v1/chongzhen/simulator

- 签名机制：接口除sign外的所有参数按参数名升序排列组合成key1=value1&key2=value2···&key做md5（key由运营提供），防止非法调用。

- 时间戳：参数time为Unix时间戳（单位：秒），超时时间2分钟，避免重复请求。

2.2 通用响应格式

所有接口响应遵循统一格式，AI相关返回结果（如人才建言、问政反馈）需包含自然语言描述，同时提供结构化数据供前端渲染，支撑AI文本反馈与游戏数据同步。

{
  "code": 0, // 状态码：0-成功，非0-失败（详见错误码表）
  "message": "操作成功", // 提示信息，失败时返回具体错误原因
  "data": {}, // 业务数据，不同接口返回不同结构，包含AI交互相关内容
  "meta": {
    "model_version": "v1.0.0", // AI模型版本
    "latency_ms": 120, // 接口响应延迟（毫秒）
    "timestamp": 1620000000 // 响应时间戳（Unix秒级）
  }
}

2.4 AI交互通用要求

- 自然语言支持：接口支持玩家输入自然语言（如问政话术、诏书内容），AI需精准解析意图，生成符合历史背景、人物性格的响应，避免现代用语，贴合明末文言风范。

- 动态推演：AI响应需结合当前游戏状态（国库、民心、边关、派系等），实现“决策-反馈”的闭环，如玩家下达诏书后，AI推演执行效果并同步更新游戏数据。

- 性格还原：AI生成的人才对话、大臣建言，需贴合人物特质（如孙承宗的稳健、徐光起的博学、魏忠贤的功利），避免同质化响应。

- 多选项支持：问政、人才建言等场景，AI需生成2-3个差异化选项，供玩家选择，每个选项对应不同的推演结果，支撑“千人千局”的玩法特性。

- 上下文记忆：AI需保留当前玩家的交互上下文（如历史问政记录、人才任用记录），确保后续响应的连贯性，避免逻辑断层，支持长上下文的连贯推演。

3.2 核心接口详细设计

3.2.1 人才招募接口（/talent/recruit）

核心功能：玩家通过招募（如科举、举荐、寻访）获取人才，AI根据当前游戏年份、朝堂态势、玩家行为，生成符合设定的人才，支持历史人物与虚构人才结合，确保人才特质的多样性与拟真性。

请求参数

{
  "player_id": "string", // 必选，玩家唯一ID
  "recruit_type": "string", // 必选，招募类型：imperial_exam（科举）、recommend（举荐）、search（寻访）
  "cost": { // 必选，招募消耗的资源
    "silver": number, // 白银数量
    "grain": number, // 粮食数量（可选，部分招募类型不需要）
    "prestige": number // 威望值（必选，崇祯核心资源，影响招募质量）
  },
  "ai_options": { // 可选，AI生成人才的配置
    "talent_quality": "string", // 可选，人才品质：ordinary（普通）、excellent（优秀）、epic（史诗，历史人物）
    "focus_field": "string", // 可选，侧重领域：military（军事）、politics（政治）、economy（经济）、culture（文化）
    "include_historical": boolean // 可选，是否包含历史人物，默认true
  },
  "time": number, // 必选，Unix时间戳（秒）
  "sign": "string" // 必选，接口签名
}

响应参数

{
  "code": 0,
  "message": "招募成功",
  "data": {
    "talent_list": [ // 招募到的人才列表（1-3人，根据招募类型而定）
      {
        "talent_id": "string", // 人才唯一ID
        "name": "string", // 人才姓名（历史人物用真实姓名，虚构人物随机生成）
        "quality": "string", // 人才品质
        "field": "string", // 擅长领域
        "ability": { // 能力值（1-100，AI根据品质、领域生成）
          "military": number, // 军事能力
          "politics": number, // 政治能力
          "economy": number, // 经济能力
          "culture": number, // 文化能力
          "loyalty": number // 忠诚度（1-100，受玩家威望、招募方式影响）
        },
        "personality": "string", // 性格（AI生成，如：刚正不阿、圆滑世故、谨慎多疑）
        "faction": "string", // 派系（AI生成，如：东林党、阉党、中立）
        "background": "string", // 背景描述（AI生成，贴合明末历史，如：“陕西举人，深谙边事，曾随孙承宗守辽东”）
        "ai_feedback": "string" // AI生成的人才表态（自然语言，如：“臣蒙陛下赏识，定当鞠躬尽瘁，死而后已”）
      }
    ],
    "cost": { // 实际消耗的资源
      "silver": number,
      "grain": number,
      "prestige": number
    },
    "current_prestige": number // 招募后玩家剩余威望值
  },
  "meta": {
    "model_version": "v1.0.0",
    "latency_ms": 150,
    "timestamp": 1620000000
  }
}

AI交互说明

- AI根据招募类型、玩家威望、当前游戏态势（如战乱时期优先生成军事人才，灾荒时期优先生成经济、民生类人才），生成人才的能力值、性格、派系，确保贴合场景需求。

- 历史人物的能力值、性格、派系固定（如孙承宗军事95、政治85，东林党，谨慎稳健；魏忠贤政治80、经济75，阉党，功利自私），AI仅生成背景描述和表态文本，还原历史人物特质。

- 虚构人才的所有属性由AI随机生成，但需符合明末历史背景（如籍贯、出身、擅长领域贴合当时社会现状），避免出现不符合时代的设定。

3.2.2 人才交互接口（/talent/interact）

核心功能：玩家通过自然语言与人才交互（如询问边关策略、安抚不满的人才、斥责失职的人才），AI根据人才性格、忠诚度、当前游戏态势，生成符合语境的响应，支持多轮对话，增强沉浸感。

请求参数

{
  "player_id": "string", // 必选，玩家唯一ID
  "talent_id": "string", // 必选，交互的人才ID

  "history": [ // 可选，历史交互记录（多轮对话时传入，确保AI上下文连贯）
    {
      "role": "player", // 角色：player（玩家）、talent（人才）
      "content": "string", // 交互内容
      "timestamp": number // 交互时间戳
    }
  ],
  "time": number, // 必选，Unix时间戳（秒）
  "sign": "string" // 必选，接口签名
}

响应参数

{
  "code": 0,
  "message": "交互成功",
  "data": {
    "talent_id": "string",
    "talent_name": "string",
    "ai_response": "string", // AI生成的人才响应（自然语言，贴合性格，如孙承宗：“陛下，臣以为当加固宁远城，启用袁崇焕，以守为攻，徐图恢复辽东”）
    "loyalty_change": number, // 交互后忠诚度变化（-5~+5，如安抚+3，斥责-4，无影响则为0）
    "attitude": "string", // 人才当前态度（AI生成，如：恭敬、不满、坚定、犹豫）
    "suggestion": { // 可选，AI生成的额外建议（若玩家询问策略时返回）
      "content": "string", // 建议内容
      "effect": "string" // 建议执行后的预期效果（AI推演，如：“若采纳此策，边关战力+10，但国库消耗白银5万两”）
    },
    "history": [ // 更新后的交互记录（包含本次交互）
      {
        "role": "player",
        "content": "string",
        "timestamp": number
      },
      {
        "role": "talent",
        "content": "string",
        "timestamp": number
      }
    ]
  },
  "meta": {
    "model_version": "v1.0.0",
    "latency_ms": 200,
    "timestamp": 1620000000
  }
}

AI交互说明

- AI需精准解析玩家输入的自然语言意图，结合人才性格、忠诚度、擅长领域，生成响应文本，避免答非所问（如询问军事策略，擅长经济的人才会表示“臣不擅军事，不敢妄言，但可保证粮草供应”）。

- 响应文本需贴合明末文言风范，避免现代用语，同时体现人物性格（如刚正不阿的人才会直言进谏，圆滑世故的人才会委婉表达），增强沉浸感。

- AI根据交互内容，推演忠诚度变化，同时生成合理的态度反馈，若玩家询问策略，AI需结合当前游戏状态（国库、边关、民心等），生成具体、可落地的建议，并提示预期效果，支撑玩家决策。

- 支持多轮对话，AI通过history参数保留上下文，确保后续响应的连贯性（如玩家先询问粮草储备，再询问赈灾策略，AI会结合前文的粮草数据给出建议）。

3.3 其他接口简要说明

- 人才详情查询接口：支持根据人才ID查询完整信息，AI生成的履历、状态评价需贴合人才的过往行为和当前态势，如“该人才任职期间，成功安抚陕西流民，民心+8，但因触动地主利益，东林党不满度上升”。

- 人才培养接口：玩家消耗资源（如白银、书籍、军功）培养人才，AI根据培养方式推演能力变化（如军事培养提升军事能力，文化培养提升文化能力），同时可能影响忠诚度（如给予重赏提升忠诚度），生成培养反馈文本。

- 人才任用接口：玩家将人才任命到指定职位，AI推演任用对对应领域的影响（如任命军事人才为兵部尚书，边关战力提升），同时生成人才任职表态，若人才派系与职位不匹配（如阉党人才任御史），会触发朝堂派系矛盾，AI生成对应事件反馈。

- 人才动态刷新接口：每回合自动调用，AI结合全局态势（如灾情、战乱、玩家决策），刷新人才状态（如战乱时期军事人才忠诚度可能上升，灾荒时期经济人才忠诚度可能下降），生成事件反馈（如“因边关告急，XX人才主动请战，忠诚度+5”）。

{

      "name": "string", // 角色名称（玩家/大臣姓名）

      "timestamp": number // 时间戳

4.3 其他接口简要说明

- 问政议题生成接口：每回合自动调用，AI结合当前游戏态势（如灾情、边关告急、国库空虚），生成1-3个待议议题，每个议题包含背景说明、当前困境，供玩家选择问政，确保议题贴合时局，支撑游戏的动态性。

- 问政结果查询接口：玩家可查询上一次问政或诏书的执行结果，AI生成详细的结果总结、数据变化、各方反馈，帮助玩家复盘决策，调整后续策略，同时保留历史记录，支撑AI的长上下文记忆。

- 派系态度查询接口：玩家可查询各派系对当前议题、诏书的态度，AI根据派系利益、议题影响，生成态度描述（如支持、反对、中立）和影响力变化，帮助玩家了解朝堂派系动态，避免因派系矛盾引发朝堂动荡，贴合明末派系斗争的历史背景。

5. AI适配补充说明

5.1 AI模型接入规范

- 接口需预留AI模型接入入口，支持替换不同的LLM模型（如Gemini、GPT等），适配游戏海外、国内版本的需求，模型调用参数（如temperature）可通过接口动态配置，控制AI响应的随机性和准确性，贴合不同玩家的玩法需求。

- AI生成的自然语言文本（人才表态、大臣建言、诏书反馈等），需经过过滤处理，避免出现不符合明末历史背景、违规、低俗的内容，同时确保文本流畅、贴合人物性格，增强沉浸感。

- AI推演逻辑需贴合明末历史设定（如官职体系、税收制度、军事制度、派系分布），避免出现不符合历史的推演结果（如出现现代官职、现代政策），确保游戏的拟真性。

- 支持AI模型版本管理，接口响应中包含模型版本信息，便于后续迭代优化，同时记录AI调用日志，用于排查问题、优化推演逻辑。

5.2 性能优化建议

- AI交互接口（人才交互、大臣建言、诏书下达）需优化响应速度，目标延迟≤500ms，可通过缓存常用AI响应模板、优化模型推理速度、分布式部署推理节点等方式实现，避免影响玩家实时交互体验。

- 批量操作接口（如人才动态刷新）可采用异步处理，避免阻塞主线程，确保游戏流畅运行，同时通过Redis缓存游戏状态数据，提升AI推演效率，减少重复计算。

- 针对长上下文交互（多轮对话、历史决策记忆），可采用上下文压缩、关键信息提取等方式，减少AI模型的输入量，提升响应速度，同时确保上下文的连贯性，避免记忆断层。

可扩展性设计

- 所有接口的请求、响应参数均预留扩展字段（如extra），支持后续新增功能（如人才技能、问政新议题、AI新交互模式），无需修改接口核心结构，降低迭代成本。

- 人才、大臣的属性（如能力值、性格、派系）支持动态扩展，可通过配置文件新增属性类型，AI推演逻辑同步适配，支撑人才玩法、问政玩法的后续迭代。

- 支持新增朝堂派系、人才类型、问政议题，接口无需大幅修改，仅需更新AI推演逻辑和配置文件，适配游戏内容的持续丰富，支撑“千人千局”的玩法升级。
