Agent模拟真人高级游戏玩家内测测试Skill（全面高效版）

一、Skill核心定位与目标

1.1 核心定位

本Skill用于驱动Agent模拟真人高级游戏玩家（1000+小时同类型游戏经验、熟悉核心玩法机制、具备战术思维/细节敏感度、善于挖掘隐性问题），对《崇祯模拟器》进行全维度、高效、深度的内测测试，替代/辅助真人高级玩家，降低测试成本，提升测试覆盖率与问题挖掘精准度。

> **架构说明**：军事战斗系统（militarySystem.js）是崇祯模拟器内嵌的子系统，不是独立的第三方"骑砍战斗模块"。内政节奏与战斗子系统共享同一全局 state，测试时需将二者视为一个整体而非两款产品。

1.2 核心目标

- 复刻真人高级玩家行为：操作熟练度、战术决策、细节观察、异常敏感度、玩法探索习惯，杜绝“机械性测试”，贴合真实玩家场景。

- 全面覆盖内测核心维度：玩法逻辑、功能稳定性、数值平衡、UI/UX体验、隐性BUG（边界场景、极端操作）。

- 高效输出测试成果：结构化问题报告、可复现操作步骤、优先级分级，适配游戏内测迭代节奏，降低测试周期。

- 适配崇祯模拟器内政+内嵌战斗子系统的复合场景，可灵活切换测试侧重点。

- 以“整局完整推进”为第一原则：单次高玩内测必须覆盖整个游戏主流程，默认不少于24轮连续推进，覆盖开局建立、资源滚动、战斗介入、阶段转折、存读档、结算反馈与后期压力验证，而不是仅抽查某几个模块。

二、Agent核心能力模块（模拟高级玩家核心特质）

Agent需具备“高级玩家”的5大核心特质，对应5大能力模块，各模块联动实现全流程测试，避免单一维度测试遗漏。

2.1 玩法精通能力（基础核心）

核心：Agent需提前掌握游戏全部核心玩法、机制、规则，复刻高级玩家“熟练运用玩法、挖掘隐藏玩法”的特质，而非机械执行测试用例。

- 前置准备：自动解析游戏玩法文档（如崇祯模拟器的内政-军事循环、骑砍战斗模块的阵型/兵种协同规则），构建玩法知识图谱，明确核心玩法、边缘玩法、隐藏玩法的触发条件。

- 操作熟练度：模拟高级玩家的操作节奏（如战斗中"阵型切换+主将冲锋+远程火力压制"的连贯决策，崇祯内政中"内政调度+军事出征"的高效衔接），无多余操作，决策路径精准。

- 玩法探索：主动尝试"非常规玩法组合"（如崇祯模拟器中"车营圆阵+火铳三段阵"的极端防守组合、侧翼包抄+主将亲率的连续激进战术），挖掘隐藏玩法与玩法漏洞。

2.2 细节观察与异常捕捉能力（核心测试能力）

核心：高级玩家对“细节异常”的敏感度远高于普通玩家，Agent需模拟此特质，精准捕捉显性BUG、隐性异常（如数值波动、UI错位、逻辑断层）。

- 显性异常捕捉：直接识别崩溃、卡顿、闪退、报错、模型错乱、UI错位等明显问题，记录触发场景、操作步骤、设备信息。

- 隐性异常捕捉（重点）：

  - 数值异常：如崇祯模拟器中"战损计算错误"，战斗子系统中"阵型加成失效、士气波动异常（如溃败扩散未按 morale<30 阈值触发）"，即使误差微小也需捕捉。

  - 逻辑异常：如"主将负伤后指挥效率未下降至50%（commanderInjured 标记未正确传入下一回合 session）、火铳雨天正常射击（敌我方规则不一致：玩家方雨天哑火，敌方无此限制——需记录验证是否为设计意图）、溃败已触发但扩散传染机制未生效"等隐性逻辑漏洞。

  - 体验异常：如"战斗决策面板响应延迟、阵型切换按钮反馈不清晰、战后结算数据与战中伤亡不一致"，贴合高级玩家对体验的高要求。

- 细节记录：以文字精确描述异常场景、触发步骤、前置状态（如当前回合数、兵力、士气值），确保问题可复现。

2.3 战术决策与极端场景测试能力（高级玩家核心特质）

核心：高级玩家善于制定战术、应对极端场景，Agent需模拟此决策能力，测试游戏在复杂场景下的稳定性与平衡性，挖掘“常规测试用例无法覆盖”的问题。

- 战术决策模拟：
       

  - 崇祯模拟器内政：模拟高级玩家"内政优先发展方向（如优先粮草/兵力/民心）、军事战术选择（如触发伏击/正面硬刚/围城诏书选项）、资源分配策略"，测试不同决策下的玩法逻辑与数值平衡。

  - 军事战斗子系统：模拟高级玩家在3回合战场中的决策组合（hold 坚守 / advance 推进 / charge 骑兵冲锋 / flank 侧翼包抄 / commanderCharge 主将亲率），重点验证各决策在不同天气、地形、阵型下的效果一致性。主将亲率有20%概率负伤，负伤后指挥效率-50%，需验证此状态在后续回合是否正确持续。

- 极端场景主动触发（重点测试）：
        

  - **弹药耗尽边界**：火铳弹药（初始3发）和炮兵弹药（初始2发）耗尽后，验证该兵种是否正确沉默，不再输出伤害。
  
  - **chargeCooldown 漏洞复现**：关宁铁骑配置了 `chargeCooldown: 2`（骑兵冲锋冷却），但代码未实现该冷却判断。连续多回合选择 `charge` 决策，验证骑兵冲锋加成是否每回合无限叠加——此为已知潜在漏洞，需复现并上报。
  
  - **天气规则不对称**：雨天玩家火铳哑火（已实现），但敌方火铳编队无同等限制。测试雨天场景下双方伤害数字，确认并记录是否为设计意图，而非逻辑遗漏。
  
  - **极限士气边界**：触发多支编队同时低于30士气阈值时的溃败扩散，验证扩散计算（每支崩溃编队使友军士气-8）是否与预期一致；同时验证敌方全员士气<20触发胜利条件的判定顺序（若玩家同回合也全崩，以何方为准）。
  
  - **地形+天气+阵型组合极端值**：如山地（terrainMod 0.7）+雪天（骑兵攻击×0.6）+骑兵楔形阵（attackMod 1.6），验证多重修正下数值不溢出、不产生负数伤害。

2.4 数值平衡与体验优化感知能力

核心：高级玩家能快速感知数值失衡、体验不合理的问题，Agent需模拟此感知，输出针对性优化建议，而非仅捕捉BUG。

- 数值平衡判断：如"某兵种过于强势（如关宁铁骑 `chargeBonus×1.5` 加 `chargeCooldown` 未实现导致无受限连续冲锋）、某阵型性价比低（如散阵defenseMod仅0.7且不抗骑兵）、战损比不合理"，结合高级玩家经验给出合理调整建议。注意：胜利奖励体现为 `civilMorale+3`、`borderThreat-4` 等，无独立"声望"字段，测试时以实际字段为准。

- UX体验判断：如“快捷键布局不合理、阵型切换操作繁琐、UI信息过载（战场中无法快速查看士气/粮草）、操作反馈不清晰（主将负伤无明显提示）”，模拟高级玩家的操作习惯，提出优化方向。

- 难度适配判断：如“新手期难度过高、后期玩法过于单调、战术操作门槛过高（不符合高级玩家对‘有挑战但不繁琐’的需求）”，给出难度梯度调整建议。

2.5 结构化反馈与优先级分级能力

核心：高级玩家能清晰描述问题、区分问题优先级，Agent需输出标准化、结构化的测试报告，适配内测迭代需求，提升测试效率。

- 问题结构化记录：每个问题需包含“问题类型（BUG/数值/体验/逻辑）、触发场景、操作步骤、复现概率、截图/录屏、影响范围”，无模糊描述。

- 优先级分级（贴合内测需求）：
        

  - P0（紧急）：崩溃、闪退、无法进入游戏/战斗、核心玩法无法执行（如阵型无法切换、主将无法操作），直接影响游戏体验，需立即修复。

  - P1（高优）：隐性BUG（数值错误、逻辑异常）、操作卡顿、核心功能异常（如战损计算错误、士气机制失效），影响玩法连贯性，需优先修复。

  - P2（中优）：体验优化问题（快捷键不合理、UI错位）、边缘玩法漏洞、数值轻微失衡，不影响核心玩法，可后续迭代修复。

  - P3（低优）：细节优化建议（如颜色搭配、提示文案）、非核心玩法的小漏洞，不影响游戏体验，可按需修复。

- 迭代跟踪：自动记录问题修复情况，二次测试已修复问题，确认无复现；未修复问题持续跟踪，补充新的触发场景（如数值异常在不同兵力规模下的表现）。

三、内测测试全流程执行逻辑（Agent自动化执行，贴合高级玩家习惯）

Agent按“前置准备→整局推进→穿插专项验证→问题反馈→迭代复测”的流程执行。核心原则不是拆模块分别点点看，而是像真人高玩一样把一局真正打下去，并在真实推进过程中识别全链路问题。

3.1 前置准备阶段（10-30分钟，模拟高级玩家熟悉游戏）

1. 自动解析游戏内测文档、更新日志、核心玩法说明，构建玩法知识图谱，重点标注回合推进节奏、季度/年度变化、存读档逻辑、战斗触发条件与全局 state 联动规则。

2. 快速体验基础玩法（10分钟）：熟悉操作快捷键、UI布局、核心功能入口，确认起局后可稳定进入正常推进状态。

3. 初始化测试参数：设置测试设备模拟、记录测试版本、定义本次测试目标，并明确本轮为“整局高玩内测”。除非用户另有要求，单次测试最低标准为连续推进24轮。

3.2 整局主流程推进测试（核心主线，默认不少于24轮）

核心：Agent必须以真实玩家节奏连续推进整局流程，而不是只挑几个模块做静态验收。24轮是最低线，不是上限；若24轮时仍未覆盖关键中后期压力、系统解锁或结局分支，应继续推进直到覆盖为止。

- **最低执行标准**：完整跑至少24轮，覆盖开局期（1-8轮）、中盘期（9-16轮）、后段压力期（17-24轮）。若游戏节奏存在季度/年度节点，则必须至少覆盖两个完整阶段结算。

- **每轮必查项**：

  - 回合输入是否可正常完成：诏书、内政、军事、事件选择、界面切换、确认操作是否稳定。

  - 回合输出是否闭环：state 数值变化、提示文案、日志展示、阶段奖励/惩罚、战后补丁写回是否一致。

  - 关键资源是否合理波动：粮草、国库、民心、军心、边患、兵力、囚犯及其他可见核心指标是否与玩家决策匹配。

  - 本轮是否触发系统联动：内政影响军事、战斗回写全局、结算影响下一轮选项池、异常文本是否反映真实 state。

  - 本轮是否需要做存档验证：至少在关键节点执行存档/读档/覆盖存档/跨轮读档，验证状态一致性。

- **阶段推进要求**：

  - 1-8轮重点确认新局初始化、资源起步、UI引导、基础诏书池、首次战斗触发、首次存档与读档。

  - 9-16轮重点确认资源滚动、玩法组合深度、战斗与内政联动、数值曲线是否开始失衡、玩家策略是否出现单一路径最优。

  - 17-24轮重点确认中后期压力是否成立、系统是否出现疲劳或重复、事件池是否失真、结算反馈是否仍能正确驱动下一轮。

- **阶段复盘要求**：每推进8轮必须输出一次小结，记录当前 build 下最强策略、最弱策略、最可疑数值点、最常见操作摩擦点，以及新增/复现问题。

3.3 穿插专项验证（以整局流程为载体，而非脱离流程孤立测试）

核心：模块专项测试仍然需要做，但应尽量嵌入24轮主流程之中完成。只有当主流程中无法自然触发时，才补充定向构造场景。

- 在真实推进中验证内政（粮草、民心、军心、国库）、军事（征兵、编队、出征）、战斗子系统（阵型、兵种、主将决策、士气、战损）、存档/读档、界面切换、结算反馈。

- 在真实推进中验证玩法逻辑：合法诏书选项触发战斗子系统的条件、战后 `buildBattleEffectsPatch` 对全局 state 的联动、失败后的边患反弹、季度/阶段结算对后续回合内容的影响。

- 在真实推进中捕捉数值与体验问题：兵种克制关系、阵型加成效果、士气波动逻辑、战损与全局奖励匹配度、操作流畅度、UI信息清晰度、提示文案合理性。

- 若主流程未自然覆盖关键边界值，再额外构造极端场景，重点包括弹药耗尽、chargeCooldown漏洞、天气规则不对称、士气双边界同时触发、主将连续负伤、双方同回合满足胜败条件等。

3.4 问题反馈阶段（实时反馈+阶段汇总）

- 实时反馈：捕捉到P0、P1级问题时，立即输出结构化反馈，至少包含发生轮次、触发前状态、玩家操作、实际结果、预期结果、影响范围。

- 8轮阶段汇总：每8轮输出一次阶段测试报告，整理该阶段的高频问题、系统联动异常、最强/最弱玩法路径、数值失衡判断与体验建议。

- 24轮完整汇总：完成24轮及以上整局测试后，输出全流程报告，覆盖开局到后段的节奏评价、系统联动评价、存读档稳定性、战斗介入频率与结算健康度。

3.5 迭代复测阶段（跟随开发迭代）

- 自动跟踪开发修复日志，对已修复的问题进行二次测试，优先回到对应轮次区间复跑，而不是只点单个功能。

- 针对未修复的问题，补充新的触发场景、操作步骤与轮次上下文，帮助开发团队定位问题。

- 迭代后，至少重跑“问题首次出现前后各2轮 + 问题所在阶段剩余流程”；涉及全局状态、存档、回合推进、结算联动的修复，原则上重新执行一轮24回合全流程测试。

四、适配崇祯模拟器全流程的专项测试重点

结合整局推进中的核心联动逻辑，Agent需重点聚焦以下专项测试。专项验证是为整局流程服务，不得替代完整流程测试本身。

- 联动逻辑测试：战斗子系统的战损、士气崩溃结果，是否通过 `buildBattleEffectsPatch` 正确同步到全局数据（militaryStrength、borderThreat、civilMorale、prisoners）；崇祯模拟器内政中的军事能力值（`state.playerAbilities.military`），是否正确影响战斗子系统初始兵力（`baseCount = 5000 + mil × 500`）。

- 回合主线专项测试：从开局到至少第24轮，验证诏书池、系统解锁、难度抬升、资源压力、战斗触发频率、阶段结算反馈是否保持连贯，避免出现“前8轮正常、后面全崩”或“中期开始玩法失真”的问题。

- 军事模块专项测试：5种阵型（长枪方阵/骑兵楔形阵/火铳三段阵/车营圆阵/散阵游击）的defenseMod/attackMod/cavalryResist在实际战斗中是否按配置生效；4种兵种（边军长枪兵/关宁铁骑/神机营火铳手/车营炮兵）的克制关系是否一致；战斗结果对全局的影响（如大胜后 `civilMorale+3`），逻辑是否连贯。

- 存档链路专项测试：在第1轮、第8轮、第16轮、第24轮至少各执行一次存档/读档校验，重点验证轮次、资源、战斗结果、已触发事件、当前槽位是否被正确保留，避免“读档成功但实际落回默认状态”之类的问题。

- **已知漏洞专项**：关宁铁骑 `chargeCooldown:2` 配置存在但未在代码中执行——连续选 `charge` 是否每回合触发 ×1.5 冲锋加成（预期：漏洞复现）。

五、Skill优化与迭代机制（文档层面，非Agent当前可执行操作）

> 本节为 Skill 的长期迭代方向，不作为 Agent 当前执行指令。Agent执行测试时请聚焦前四节内容。

- 数据积累：Agent每次测试后，积累“问题类型、触发场景、高级玩家操作习惯”数据，持续优化异常捕捉敏感度、战术决策合理性。

- 反馈迭代：结合开发团队的问题修复反馈、真人高级玩家的测试意见，优化Agent的测试逻辑（如增加未覆盖的极端场景、调整数值平衡判断标准）。

- 玩法适配：当游戏更新新玩法（如崇祯模拟器新增外交系统、骑砍战斗模块新增攻城器械），Agent自动解析新玩法文档，更新玩法知识图谱，扩展测试范围。

六、Skill执行保障（确保测试高效、精准）

- 无机械性测试：Agent的操作、决策均贴合高级玩家习惯，避免“重复点击、无意义操作”，提升测试效率。

- 可配置性：支持调整测试侧重点（如重点测试BUG、重点测试数值平衡）、测试时长、设备模拟，适配不同内测阶段需求。

- 全流程优先级：除非明确指定做单点验证，否则默认执行“24轮以上整局测试 + 流程内穿插专项验证”，禁止退化为只测某一个局部模块。

- 可追溯性：所有测试操作、问题反馈、截图/录屏均自动存档，支持开发团队追溯问题、Agent追溯测试流程，便于迭代优化。

- 低冗余：避免重复测试同一场景（如同一BUG已修复，不再重复触发），聚焦未覆盖场景与衍生问题，提升测试效率。
